Frontend is served at `http://127.0.0.1:8090` by `invisible-frontend`, which is a DIFFERENT origin — fetch from the bubble crosses ports. Backend contract (mirror of 01-01-SUMMARY.md; if SUMMARY is missing or differs, the SUMMARY wins — re-read it):

Request:
- URL: `http://127.0.0.1:8765/api/v1/chat` — absolute, because the frontend is on :8090 in dev.
- Headers: `content-type: application/json`
- Body: `{"message": string, "page_context": string, "project_id"?: string}`. `project_id` is optional — omit it; today the prop signature is `<AIBubble pageContext={...} />` so only `page_context` is available.

Response (success, 200):
```
{ "text": "<assistant reply prose>", "usage": { "input_tokens": int, "output_tokens": int, "cost_usd": float, "duration_ms": int }, "cost": float }
```

Response (errors — all return JSON `{ "error": "<code>", "hint": "<one-line remediation>" }`):
- 400 bad_request → "Couldn't send: {hint}"
- 401 claude_unauthenticated → "Claude CLI not signed in. {hint}" (hint will read "run: claude login")
- 413 message_too_large → "Message too long (max 8000 characters)."
- 429 rate_limited → "Claude rate-limited. {hint}"
- 502 claude_cli_failed → "Claude CLI failed. {hint}"
- 504 timeout → "Claude took too long (>60s). Try a shorter question."

Network/fetch-rejected (no HTTP response at all — e.g., backend daemon not running): caught in the `try/catch`. Render: "Backend unreachable — is `invisible-dashboard` running on 127.0.0.1:8765?"

Conversation history scope: REQ-02 specifies "in-memory; not durable yet". The existing component already keeps `msgs` in `useStateAI`. That's the page-session list. Do NOT add localStorage. Do NOT send prior messages to the backend in this phase (backend is stateless per 01-01; future phases can layer on multi-turn context).

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser (bubble) → dashboard daemon | Cross-port loopback request from :8090 to :8765. Both bound to 127.0.0.1 in dev. With `--no-auth` (the dev mode), no bearer token is required; the LAN-binding refusal in the daemon prevents this from being remote-exploitable. |
| user input → backend prompt | User types arbitrary text into the bubble input. The frontend sends it untouched to the backend. Backend (Plan 01-01) is responsible for not letting that string reach a shell. Frontend's responsibility is only to bound the size client-side. |
| backend error `hint` → bubble DOM | The `hint` field from the backend is interpolated into a React text node (NOT `dangerouslySetInnerHTML`), so React's default escaping prevents XSS even if a future regression in `_redact` leaks raw HTML. |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-09 | Tampering (XSS) | bubble error renderer | mitigate (medium) | All response strings are rendered through React text children (`{m.text}`), which auto-escapes. No `dangerouslySetInnerHTML` introduced. Verified by absence in acceptance criteria. |
| T-01-10 | DoS | bubble input | mitigate (medium) | Client-side cap at MAX_MESSAGE_CHARS = 8000 short-circuits before fetch, mirroring backend's HTTP 413. |
| T-01-11 | Information Disclosure | error UI | mitigate (low) | The bubble shows only the backend's already-redacted `hint`. No browser console.log of credentials or environment data is added. |
| T-01-12 | Repudiation | conversation history | accept | History is intentionally in-memory only per REQ-02. No persistence means no audit trail; acceptable for Phase 1 demo scope. Revisit when chat is wired to durable storage. |
| T-01-13 | Tampering | npm/pip/cargo installs | mitigate | No new dependencies. Babel-standalone + React globals only. No package legitimacy gate required. |
| T-01-14 | Elevation of Privilege | bubble bypassing backend validation | accept | Client-side size cap is defense-in-depth; backend independently enforces 8000-char limit + JSON shape validation. A malicious bubble (modified DOM) cannot escalate beyond what backend HTTP validation allows. |
</threat_model>
