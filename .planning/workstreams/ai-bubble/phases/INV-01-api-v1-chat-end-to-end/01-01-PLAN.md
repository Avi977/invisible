<interfaces>
Canonical CLI invocation pattern lives in `lib/runners.py`. Mirror it. Key bits:

From lib/runners.py:
- `CONFIG["claude_cmd"] = ["claude", "-p", "--output-format", "json"]` — exact argv. Do NOT use shell=True.
- `_run(cmd, cwd, stdin, timeout)` — wraps subprocess.run with `capture_output=True, text=True, timeout=timeout, check=False`. Catches `subprocess.TimeoutExpired` and `FileNotFoundError`.
- `_extract_claude_usage(raw)` — pulls `{input_tokens, output_tokens, cache_*, cost_usd, duration_ms}` from the JSON envelope's `usage` and `total_cost_usd` fields. Returns None if envelope malformed.
- `_parse_claude_json(raw)` — claude's `--output-format json` envelope wraps the actual reply under `result`. The reply itself may or may not be JSON. For chat, treat `result` as free-text reply, NOT as inner-JSON to re-parse.

Dashboard handler pattern, from bin/invisible-dashboard:
- `class DashboardHandler(BaseHTTPRequestHandler)` — currently has `do_GET` only.
- `self._send_json(obj, status)` — emits Content-Type application/json + Content-Length headers. Reuse for /api/v1/chat responses.
- Routing is inline if-blocks inside `do_GET`. Add a `do_POST` with the same shape.
- Auth check: `if not self._auth_ok(): return 401`. Apply this to POST too. With `--no-auth` it short-circuits to true (this is the dev mode the bubble runs in).
- `self.headers.get("Content-Length")` and `self.rfile.read(n)` are the stdlib idioms for reading a POST body in BaseHTTPRequestHandler.

JSON response contract (this plan defines it; frontend plan consumes it):
- Success (200): `{"text": "<assistant reply as plain string>", "usage": {"input_tokens": int, "output_tokens": int, "cost_usd": float, "duration_ms": int}, "cost": float}`. `cost` is a top-level convenience copy of `usage.cost_usd`.
- Bad request (400): `{"error": "bad_request", "hint": "<what was wrong>"}`. Triggered by: malformed JSON body, missing `message`, missing `page_context`, non-string fields.
- Payload too large (413): `{"error": "message_too_large", "hint": "max 8000 chars"}`.
- Claude not authenticated (401): `{"error": "claude_unauthenticated", "hint": "run: claude login"}`. Detect by probing `claude auth status` OR by inspecting CLI exit code + stderr for "not logged in"/"authentication" tokens.
- Rate-limited (429): `{"error": "rate_limited", "hint": "claude API rate limit; wait a minute"}`. Detect by stderr/stdout containing "rate limit"/"429"/"too many requests" (case-insensitive).
- Timeout (504): `{"error": "timeout", "hint": "claude CLI exceeded 60s"}`.
- CLI binary missing or non-zero with no recognized signal (502): `{"error": "claude_cli_failed", "hint": "<stderr-redacted: first 200 chars with absolute paths stripped>"}`.

In-memory history scope: REQ-02 says "Conversation history persists for the page session (in-memory; not durable yet)." Per phase guidance, frontend owns the page-session list; backend is stateless for this phase. Do NOT add a server-side conversation store. Do NOT accept a `history` field yet (frontend plan will replay the system+prior-turns into the `message` payload if it needs them — that's its choice). This keeps the backend trivially safe to restart.

| Boundary | Description |
|----------|-------------|
| client → dashboard daemon | Loopback-only by default; `--no-auth` is allowed only when bound to 127.0.0.1. With auth on, a bearer token gates POST as well as GET. Bubble runs in `--no-auth` localhost mode. |
| dashboard daemon → claude CLI | Untrusted user `message`/`page_context`/`project_id` strings flow into the prompt fed via subprocess STDIN. Never the argv. |
| claude CLI → external Anthropic API | Out of scope (Anthropic's surface). |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-01-01 | Tampering / Injection | `lib/api/chat.py` subprocess invocation | mitigate (high) | argv-list invocation only; constant `CLAUDE_CMD = ["claude", "-p", "--output-format", "json"]`; user input passed via `subprocess.run(..., input=prompt, ...)` STDIN; no `shell=True`; lint/grep gate in acceptance ensures `shell=True` count == 0. |
| T-01-02 | DoS | `lib/api/chat.py` message validation | mitigate (high) | Reject `message` >8000 chars with HTTP 413; reject body >32_768 bytes at the dashboard layer before JSON parse. |
| T-01-03 | DoS | `lib/api/chat.py` subprocess wall-clock | mitigate (high) | Hard 60s timeout; `subprocess.TimeoutExpired` → HTTP 504 (kills the child). |
| T-01-04 | Information Disclosure | error response bodies | mitigate (medium) | Stderr is redacted with regex `r"/[^\s]+"` → `<path>` and truncated to 200 chars before going into `hint`. Stack traces print to stderr only, never to the response. |
| T-01-05 | Information Disclosure | cross-project leakage via `project_id` / `page_context` | mitigate (high) | This phase treats both as opaque labels — interpolated into the prompt STRING for context, never used to construct filesystem paths or shell out to anything project-scoped. | T-01-07 | Repudiation / Audit | no request log for chat | accept | Phase 1 is read-mostly demo. Acceptable to skip request audit logging; the orchestrator's Notion review log captures user-impact events at a coarser grain. Revisit when chat persists durably. |
