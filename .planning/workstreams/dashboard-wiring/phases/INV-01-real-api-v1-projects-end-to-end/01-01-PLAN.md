Worktree layout (from bin/invisible-dashboard.list_projects):
- Checkpoints live at `home() / "worktrees" / <project> / "feature" / ".invisible-checkpoint.json"`
- The `repo_path` in invisible.toml points to the *user's* project root (e.g. `~/Projects/jobslayer`) — that's where git lives
- The orchestrator worktree (at `worktrees/<project>/feature/`) is where the checkpoint sits
- So: branch + lastCommit come from `repo_path`; status + todos + note + progress come from the orchestrator worktree's checkpoint

Status derivation (deterministic):
- checkpoint missing AND repo_path missing → "planning"
- checkpoint missing AND repo_path exists → "planning"
- checkpoint.last_verdict == "approve" AND iteration >= max_iters → "shipped"
- checkpoint.last_verdict == "changes" AND no progress in 24h → "blocked"
- otherwise → "in-progress"

Color palette (assign by stable hash of project name into this list):
["#f5b343", "#5cc8ff", "#b794ff", "#4ade80", "#f56fb1", "#5ee0c8"]

CORS:
- bin/invisible-frontend already sets `Access-Control-Allow-Origin: *` on its OWN responses,
  but that's the wrong direction — the browser enforces CORS on the *target* of the fetch.
- The dashboard daemon at 8765 must add `Access-Control-Allow-Origin: http://127.0.0.1:8090`
  to JSON responses so `fetch('http://127.0.0.1:8765/api/v1/projects')` from the React app at 8090 succeeds.
- Allowed origin should be loopback-only: only echo back when Origin starts with `http://127.0.0.1:` or `http://localhost:`.
  Do NOT use `*` — the dashboard sometimes runs without auth, and `*` would let any web page on the machine read it.

<threat_model>
## Trust Boundaries

| Boundary | Description |
|----------|-------------|
| browser (8090) → dashboard daemon (8765) | Cross-origin fetch over loopback; CORS-controlled |
| dashboard daemon → invisible.toml | File read; user-controlled paths |
| dashboard daemon → user's project repos (repo_path) | Subprocess: `git -C <path>`; path injection risk |
| dashboard daemon → orchestrator worktree | File read of `.invisible-checkpoint.json`; JSON parse risk |
| dashboard daemon → Notion API | Outbound HTTPS with bearer token from env |

## STRIDE Threat Register

| Threat ID | Category | Component | Disposition | Mitigation Plan |
|-----------|----------|-----------|-------------|-----------------|
| T-INV-01-01 | Tampering | `git -C <repo_path>` subprocess via `_branch_and_last_commit` | mitigate | Use `subprocess.run([...], shell=False, timeout=2, capture_output=True)` — array form prevents shell injection even if repo_path contains spaces or shell metacharacters. Wrap in try/except; never raise to handler. |
| T-INV-01-02 | Information Disclosure | Error responses leak absolute filesystem paths (e.g. "/Users/ace/Projects/...") | mitigate | `handle_projects()` wraps `build_projects()` in try/except; on exception returns `{"error":"internal error"}` 500 only — no traceback, no path. Test 6 asserts no home-directory substring in any response field. |
| T-INV-01-03 | Information Disclosure | Path traversal: invisible.toml repo_path = "../../../etc" reads outside intended scope | mitigate | `_safe_path()` resolves and validates the path is inside the user's home OR INVISIBLE_HOME; rejects others by returning None (treated as "repo missing"). git subprocess is read-only (`rev-parse`, `log -1`) so even on bypass no writes are possible. |
| T-INV-01-04 | Spoofing | Cross-origin fetch from a malicious local web page reads user's project list | mitigate | CORS header `Access-Control-Allow-Origin` is set ONLY for loopback origins (`http://127.0.0.1:*`, `http://localhost:*`); never `*`. Combined with the daemon's `--host 127.0.0.1` binding, only loopback web pages can read the API. |
| T-INV-01-05 | Denial of Service | Slow / hung git on a corrupt or NFS-mounted repo blocks the whole HTTP response | mitigate | `subprocess.run(..., timeout=2)` per git call; on timeout fall back to ("—","—"). Worst-case latency: N projects × 2s; acceptable for project counts < ~10. Document in code comment. |
| T-INV-01-06 | Tampering | Malicious `.invisible-checkpoint.json` (e.g. arbitrary deeply-nested JSON) crashes the JSON decoder or exhausts memory | accept | `json.loads` from Python stdlib handles deeply-nested JSON without crashing within ~1000 levels; checkpoint files are written by our own orchestrator. Risk accepted; revisit if checkpoint becomes user-editable. |
| T-INV-01-07 | Information Disclosure | Notion review titles surface internal project names to a cross-origin attacker | accept | The Notion review pull is gated by the same loopback CORS rule above. The data flows browser ↔ loopback dashboard only; Notion data does not flow outbound from the dashboard daemon. Risk accepted. |
| T-INV-01-08 | Repudiation | No audit trail of who fetched the projects endpoint | accept | Single-user tool, loopback-only binding, dashboard already swallows access logs via `log_message` no-op. Audit is out of scope per PROJECT.md "Single user". |
| T-INV-01-SC | Tampering | npm/pip/cargo installs | accept | No new package installs in this plan — pure stdlib (subprocess, hashlib, json, pathlib, urllib). pytest is already a dev-dependency. No supply-chain surface added. |
</threat_model>
